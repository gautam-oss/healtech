Approach 4 Variant

Docker + Poetry
Without virtualenv

A detailed guide to running Django inside Docker using Poetry for dependency management, with virtualenvs.create false — letting the container itself serve as the isolation layer.

Docker Compose Poetry 2.x Django 5+ No venv inside container
01 / Concept

What is this approach & why use it?

In a standard Python project, you'd create a virtual environment to isolate dependencies. But when your project already runs inside a Docker container, that container is the isolated environment — it has its own filesystem, Python binary, and packages. Adding a venv inside it adds friction with no real benefit.

This approach tells Poetry: "don't create a venv — just install packages directly into the system Python inside this container." The result is a simpler, faster, more ergonomic workflow.

🖥
Host Machine (WSL / Linux) Ubuntu, macOS, Windows
🐳
Docker Container (python:3.12-slim) OS isolation
⛔   NO virtualenv created — Poetry installs directly into system Python
🐍
System Python 3.12 + Poetry-installed packages /usr/local/lib/python3.12/
⚡
Your Django Application /home/user/app/backend/
💡
Key insight: Docker containers are immutable by design. Each container gets its own isolated Python environment from the image. There's no risk of package conflicts between projects — each project has its own image.

04 / Poetry Config

pyproject.toml — dependency declaration

Poetry replaces requirements.txt with a structured pyproject.toml. You declare dependencies with version constraints and Poetry resolves and locks exact versions in poetry.lock.

pyproject.toml TOML 
# Project metadata
[tool.poetry]
name = "myproject"
version = "0.1.0"
description = "Django project"
package-mode = false    # not a Python package, just an app

# Runtime dependencies (go into production image)
[tool.poetry.dependencies]
python = "^3.12"
Django = "^5.2"
djangorestframework = "^3.16"
psycopg = "^3.3"               # PostgreSQL driver
celery = "^5.6"
gunicorn = "^25.1"
python-decouple = "^3.8"       # reads .env files
dj-database-url = "^3.1"       # DATABASE_URL parsing
whitenoise = "^6.12"           # static file serving
sentry-sdk = "^2.54"

# Dev-only dependencies (only in dev image, excluded from prod)
[tool.poetry.group.dev.dependencies]
coverage = "^7.13"
model-bakery = "^1.23"          # test factories
pre-commit = "^4.5"
ruff = "^0.15"                  # linting + formatting

[build-system]
requires = ["poetry-core>=1.2.0"]
build-backend = "poetry.core.masonry.api"

Key Poetry commands

poetry commands bash 
# Add a new dependency
poetry add djangorestframework

# Add dev-only dependency
poetry add --group dev pytest-django

# Remove a dependency
poetry remove requests

# Update all dependencies
poetry update

# Update one package
poetry update django

# Show what's installed
poetry show

# Show dependency tree
poetry show --tree
05 / Dockerfile

backend/Dockerfile — annotated

backend/Dockerfile dockerfile 
# Start from official slim Python image (smaller than full)
FROM python:3.12-slim

# Python environment tuning
ENV PYTHONFAULTHANDLER=1 \
  # Show crash tracebacks on segfault
  PYTHONUNBUFFERED=1 \
  # Don't buffer stdout/stderr — logs appear immediately
  PYTHONHASHSEED=random \
  # Randomise hash seed for security
  PIP_NO_CACHE_DIR=off \
  # Don't cache pip downloads (keeps image lean)
  PIP_DISABLE_PIP_VERSION_CHECK=on \
  PIP_DEFAULT_TIMEOUT=100 \
  POETRY_VERSION=2.0.1
  # Pin Poetry version for reproducibility

# Create a non-root user for security
RUN groupadd user && useradd --create-home --home-dir /home/user -g user user

# Install system build dependencies
# python3-dev + gcc = compile C extensions (e.g. psycopg)
# libpq-dev = PostgreSQL headers needed by psycopg
RUN apt-get update && apt-get install python3-dev gcc build-essential libpq-dev -y

# Install Poetry via pip
RUN pip install "poetry==$POETRY_VERSION"

# Copy dependency files FIRST (before app code)
# This is a Docker layer caching trick:
# if pyproject.toml/poetry.lock don't change, Docker reuses
# the cached "poetry install" layer — much faster rebuilds
COPY pyproject.toml /home/user/app/
COPY *poetry.lock /home/user/app/

WORKDIR /home/user/app/

# ╔══════════════════════════════════════════════════╗
# ║  THE CRITICAL LINE — disables venv creation     ║
# ║  Packages install directly into system Python   ║
# ╚══════════════════════════════════════════════════╝
RUN poetry config virtualenvs.create false

# Install all deps including dev group
# --no-root: don't try to install the project itself as a package
# --no-interaction: don't ask questions
# --no-ansi: clean output in CI/build logs
RUN poetry install --with dev --no-root --no-interaction --no-ansi

# Now copy application code
# (done after install so code changes don't bust the dep cache)
WORKDIR /home/user/app/backend
COPY backend/ /home/user/app/backend

# Switch to non-root user
USER user

# Start Django via gunicorn with hot-reload for development
CMD gunicorn healtech.wsgi --log-file - -b 0.0.0.0:8000 --reload

Why copy pyproject.toml before app code?

1
Docker reads your Dockerfile top-to-bottom

Each instruction creates a new layer. Layers are cached based on content changes.

2
COPY pyproject.toml → RUN poetry install

If pyproject.toml and poetry.lock didn't change, Docker uses the cached install layer. This is a multi-minute operation — you don't want to redo it.

3
COPY backend/ (app code)

Your Python files change constantly. Placing this after poetry install means code changes don't invalidate the dependency cache.

4
Result: fast rebuilds

When you edit views.py, Docker only re-runs the COPY step — not the 3-minute poetry install.

06 / Docker Compose

docker-compose.yml — all services

The Compose file wires together all services your Django app depends on:

db
postgres:alpine
PostgreSQL database. Persisted via named volume.
broker
rabbitmq:alpine
Message broker for Celery tasks.
result
redis:alpine
Celery result backend. Also used by django-defender.
backend
./backend/Dockerfile
Your Django app. Mounts code as volume for live reload.
frontend
./frontend/Dockerfile
Webpack dev server. Serves JS bundles to Django.
celery
./backend/Dockerfile
Same image as backend, different CMD for worker.
mailhog
mailhog/mailhog
Fake SMTP server. Catch all dev emails in browser.
docker-compose.yml yaml 
services:

  # PostgreSQL — persisted via named volume
  db:
    image: "postgres:alpine"
    environment:
      - POSTGRES_USER=myapp
      - POSTGRES_PASSWORD=password
      - POSTGRES_DB=myapp
    ports:
      - "5432"    # only exposed to other containers, not the host
    volumes:
      - dbdata:/var/lib/postgresql/data:delegated

  # RabbitMQ — Celery message broker
  broker:
    image: "rabbitmq:alpine"

  # Redis — Celery result backend + django-defender cache
  result:
    image: "redis:alpine"
    ports:
      - "6379:6379"

  # Webpack dev server — serves JS bundles to Django template
  frontend:
    build:
      dockerfile: frontend/Dockerfile
      context: .
    volumes:
      - .:/app/
      - /app/node_modules   # anonymous volume: prevents host from overwriting node_modules
    ports:
      - "3000:3000"

  # Django application
  backend:
    build:
      dockerfile: backend/Dockerfile
      context: .              # context is root so Dockerfile can COPY pyproject.toml
    ports:
      - "8000:8000"
    volumes:
      - ./:/home/user/app/  # mount entire project — enables live code reload
    env_file: backend/.env
    depends_on:
      - db
      - broker
      - result
      - frontend

  # Celery worker — same image, different command
  celery:
    build:
      dockerfile: backend/Dockerfile
      context: .
    command: celery --app=myapp worker --loglevel=info
    volumes:
      - ./:/home/user/app/
    env_file: backend/.env
    depends_on:
      - db
      - broker
      - result

  # MailHog — catch all emails in dev, no real sending
  mailhog:
    image: mailhog/mailhog
    ports:
      - '1025:1025'   # SMTP port Django sends to
      - '8025:8025'   # Web UI at http://localhost:8025

volumes:
  dbdata:
    name: myapp_dbdata
    external: true  # must create manually: docker volume create myapp_dbdata
07 / Environment

backend/.env — config & secrets

The .env file is loaded by Docker Compose via env_file and read inside Django via python-decouple. It's git-ignored — each developer has their own local copy.

backend/.env env 
# Tell Django which settings module to use
DJANGO_SETTINGS_MODULE=myapp.settings.local

# Celery: use RabbitMQ as broker (preferred over Redis)
CELERY_BROKER_URL=amqp://broker:5672//
# "broker" is the Docker Compose service name — auto-resolved

# Celery: store task results in Redis
REDIS_URL=redis://result:6379
# "result" is the Redis service name

# Database — points to the "db" Compose service
DATABASE_URL=postgres://myapp:password@db:5432/myapp
# format: postgres://USER:PASS@HOST:PORT/DBNAME
🔑
Service DNS magic: Inside a Docker Compose network, service names (db, broker, result) resolve as hostnames automatically. Your Django app connects to db:5432 and Docker routes it to the postgres container. No IP addresses needed.
backend/.env.example (commit this one) env 
DJANGO_SETTINGS_MODULE=myapp.settings.local
CELERY_BROKER_URL=amqp://broker:5672//
REDIS_URL=redis://result:6379
# Uncomment one:
# DATABASE_URL=postgres://myapp:password@db:5432/myapp
# DATABASE_URL=sqlite:///db.sqlite3
08 / Setup Workflow

First-time project setup — step by step

1

Clone the repo & create your .env

terminal bash 
git clone git@github.com:yourorg/myproject.git
cd myproject
cp backend/.env.example backend/.env
# Open .env and uncomment DATABASE_URL=postgres://...
2

Create the external database volume

terminal bash 
docker volume create myapp_dbdata
# This persists your database across container restarts
# "external: true" in compose.yml means Compose won't auto-delete it
3

Build the images

terminal bash 
docker compose build --no-cache backend frontend
# --no-cache: forces clean build, skips any stale layers
# This is where Poetry downloads and installs all packages
# First build takes 3–5 minutes, subsequent ones are cached
4

Generate the OpenAPI schema (if using drf-spectacular)

terminal bash 
# Run management command inside the backend container (one-shot)
docker compose run --rm backend python manage.py spectacular \
  --color --file schema.yml
5

Create migrations and migrate

terminal bash 
# Create initial migrations for the users app
docker compose run --rm backend python manage.py makemigrations

# Apply all migrations to the database
docker compose run --rm backend python manage.py migrate
6

Start all services

terminal bash 
docker compose up -d
# -d: detached mode (runs in background)

# Access your app at:
# Django:   http://localhost:8000
# MailHog:  http://localhost:8025
# Redis:    localhost:6379
09 / Daily Commands

Day-to-day development workflow

Running management commands

management commands bash 
# Make migrations after editing models
docker compose run --rm backend python manage.py makemigrations

# Apply migrations
docker compose run --rm backend python manage.py migrate

# Create a superuser
docker compose run --rm backend python manage.py createsuperuser

# Collect static files
docker compose run --rm backend python manage.py collectstatic --noinput

# Open Django shell
docker compose run --rm backend python manage.py shell

# The Makefile shortcuts (from the HealTech project)
make docker_makemigrations
make docker_migrate
make docker_backend_shell

Running tests

testing bash 
# Run all tests
docker compose run --rm backend python manage.py test

# Run a specific test file
docker compose run --rm backend python manage.py test users.tests.test_views

# Run with parallel execution + keep DB between runs (faster)
docker compose run --rm backend python manage.py test --parallel --keepdb

# Run with coverage (ruff linter is also installed in dev group)
docker compose run --rm backend coverage run manage.py test

Logs & debugging

logs & shell access bash 
# Stream logs from all services
docker compose logs -f

# Stream logs from one service only
docker compose logs -f backend
docker compose logs -f celery

# Open a bash shell inside the running backend container
docker compose exec backend bash

# Or run a one-off bash shell
docker compose run --rm backend bash

# Inside the shell, Poetry commands work normally:
poetry show            # list installed packages
poetry add requests    # add a new package (rebuilds needed after)
python -c "import django; print(django.__version__)"

Adding a new Python dependency

adding dependencies bash 
# 1. Open a shell inside the backend container
docker compose run --rm backend bash

# 2. Inside the container, add the dependency
poetry add requests

# 3. Exit the shell
exit

# 4. Rebuild containers to install the new package into the image
docker compose down
docker compose up -d --build

# ─── OR use the Makefile shortcut ───────────────────
make docker_update_dependencies
⚠️
Important: When you add a dependency with poetry add inside a container, it updates pyproject.toml and poetry.lock on your host (because the project directory is volume-mounted). But other containers won't have the new package until you rebuild with docker compose up --build.
10 / Trade-offs

Pros and cons of this approach

✓ Pros

  • No source .venv/bin/activate ceremony ever
  • Poetry handles dev/prod dependency groups cleanly
  • Exact lock file means 100% reproducible builds across machines and CI
  • New developer onboarding is just docker compose up
  • All services (DB, Redis, RabbitMQ, MailHog) spin up together
  • postgres, redis versions pinned — no "works on my machine"
  • Running python manage.py feels natural inside container
  • Same image runs in CI/CD — eliminates environment drift

✕ Cons

  • Rebuild required after every dependency change
  • First build is slow (3–5 min) until caches warm up
  • File system performance on WSL2 volume mounts can be sluggish
  • Every command is prefixed with docker compose run --rm backend
  • Debugging is harder — extra indirection to get inside the container
  • No IDE autocomplete without pointing IDE at container Python
  • Poetry itself runs inside container — can't use it from host directly
  • Docker must be running before any work starts
11 / Decision

When to choose this approach

✅
Choose this when: You're building a real product with a team. Your project has multiple services (Django + Postgres + Redis + Celery + RabbitMQ). You want the local dev environment to mirror production as closely as possible. Team members have different OS setups (WSL, macOS, Linux). CI/CD runs the same image.
✕
Don't choose this when: You're learning Django for the first time. You're building a quick proof-of-concept that'll be thrown away. You work alone on a laptop where the friction isn't justified. Docker Desktop is unavailable or restricted.
🎯
Real-world fit: This is exactly what projects like HealTech v2 and MediCore need. 7-service architectures with RabbitMQ, Redis, and multiple Django workers benefit enormously from docker compose up being the only setup command.
